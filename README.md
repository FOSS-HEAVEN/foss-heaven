## ⚠️ What This Is NOT

Before you get excited, understand what you're looking at:

- ❌ **A business.** There is no company, no LLC, no "team." Just people who deploy software.
- ❌ **A monetization play.** No affiliate links. No sponsors. No "partnerships." 
- ❌ **A growth hack.** No email gates. No "lead magnets." No Twitter threads with 47 slides.
- ❌ **Comprehensive.** If a tool isn't here, we probably haven't used it. That's okay.
- ❌ **Permanent.** Licenses change. Projects die. Maintainers burn out. We update when we can.
- ❌ **A replacement for your own judgment.** Verify everything. Trust nothing blindly.

## 🎯 Project Objectives

### 1. Tell the Truth About Software
Most "reviews" are SEO content written by people who never ran the software. We only list tools we've deployed, broken, fixed, or abandoned. If something sucks, we say so. If something is great but has a fatal flaw, we mention the flaw.

### 2. Respect the Reader's Time
No clickbait. No "you won't believe #7" lists. No 5,000-word introductions before getting to the point. Each entry tells you what the tool does, why it matters, and whether it's worth your time.

### 3. Stay Current (Within Reason)
Open source moves fast. Licenses change. Forks happen. Maintainers ghost. We remove dead projects. We flag license changes. We note when a once-great tool has become abandonware.

### 4. Zero Commercial Influence
No vendor gets preferential treatment because they have a marketing budget. No affiliate links. No sponsored entries. If a tool is listed, it's because someone actually uses it, not because someone paid for placement.

### 5. Lower the Barrier to Self-Hosting
The biggest obstacle to FOSS adoption isn't the software — it's knowing where to start. We include deployment scripts, Docker Compose files, and honest assessments of complexity. If a tool requires 47 microservices to run "hello world," we say so.

## 📜 Philosophy

### On Honesty
> "If you can't say something sucks, you can't say something is great."

Every tool has tradeoffs. We don't do "everything is awesome" curation. If a tool is listed as "Industry Standard," that doesn't mean it's perfect — it means enough people use it that you should know about it.

### On Completeness
> "A complete list of all open source tools would be infinite and useless."

We'd rather have 200 verified tools than 2,000 unchecked links. Quality over quantity. Always.

### On Permanence
> "All software is temporary. All licenses are negotiable. All maintainers are human."

Projects die. Companies acquire open source projects and ruin them. Don't get attached to tools — get attached to principles.

### On Community
> "This is not a community. This is a repo."

We don't have a Discord. We don't have a forum. We don't have "community managers." We have GitHub issues and pull requests. That's enough.

## 🤝 How to Contribute

### You Can:

- **Open an issue** if a tool is dead, changed license, or is misrepresented
- **Submit a PR** with tools you've actually deployed and can vouch for
- **Call out bullshit** if something in these reports is wrong or misleading
- **Translate** the reports into other languages
- **Fix typos, broken links, or outdated info**
- **Add deployment notes** for tools you've actually run in production

### You Cannot:

- Submit tools you haven't used ("I heard it's good" is not good enough)
- Add affiliate links, referral codes, or sponsored entries
- Submit proprietary software or "open core" tools with paid feature gates
- Write marketing copy instead of honest assessments
- Demand updates on a schedule (we have jobs)

### The Golden Rule:

> **Deploy it. Break it. Fix it. Then tell us about it.**

If you haven't spent at least 2 hours with a tool — installing it, configuring it, hitting edge cases — you don't know enough to recommend it. Don't waste our time with README-driven recommendations.

## 📋 Submission Template

If you're adding a new tool, use this format:

```markdown
### Tool Name
- **License:** [OSI-approved license]
- **Language:** [Primary language]
- **Category:** [One of the 10 categories]
- **Complexity:** [Beginner / Intermediate / Advanced / Masochist]
- **What it does:** [One sentence]
- **Why it matters:** [One sentence]
- **What sucks:** [Honest criticism, required]
- **Deployment notes:** [Your actual experience, not the README]
- **Verified by:** [@your-github-handle]
- **Last verified:** [YYYY-MM-DD]
```

**No "what sucks" = no merge.** Every tool has problems. If you can't name one, you haven't used it enough.
